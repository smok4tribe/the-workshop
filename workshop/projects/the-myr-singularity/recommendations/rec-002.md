# First External Proposed Recommendation Candidate Set - rec-002

## What this file is

This document (with its machine-readable companion `rec-002.json`) records
the first **external** proposed recommendation candidate set for The Myr
Singularity. Unlike rec-001, which is limited to cards already in the deck's
Card Facts, this candidate set references external candidate Card Facts
stored in `workshop/card-data/candidate_cards.json`.

Every candidate is `proposed`, non-actionable, and undecided. These are not
approved recommendations and they are not deck changes. **No deck change is
authorized by this file.** A Product Owner decision is required before any
candidate advances, and any future accepted candidate additionally requires
a decision log entry and a new deck version file.

The referenced candidate Card Facts remain **facts-only**: referencing a
card here does not change its status in `candidate_cards.json`, and nothing
in the facts layer becomes a recommendation.

rec-001 remains the internal/current-card-facts candidate set and is
unchanged by this file.

## Reference conventions

rec-002 uses qualified references only:

- `deck:scryfall:<id>` — resolves against `workshop/card-data/cards.json`.
- `candidate:scryfall:<id>` — resolves against
  `workshop/card-data/candidate_cards.json`, must not exist in `cards.json`,
  and its fact record must remain `facts_only`.
- `incoming_cards` — cards that would enter the deck if a candidate were
  later accepted (`candidate:scryfall:<id>` only).
- `outgoing_cards` — cards that would leave the deck (`deck:scryfall:<id>`
  only, playable v1.0 cards only). All candidates in this set leave
  `outgoing_cards` empty: nominating cut targets is deliberately deferred to
  Product Owner review.
- `affected_cards` — every card a candidate touches, including all
  incoming and outgoing references.

Legacy bare `scryfall:<id>` references are reserved for rec-001 and do not
appear in this set. Card names appear in these human-readable summaries;
machine-readable references are qualified IDs only.

## Candidate lifecycle

```text
proposed -> under_review -> accepted | rejected | deferred | needs_testing
                ^                                    |
                +------------ with new evidence -----+
```

`accepted` and `rejected` are terminal and require `user_decision` and
`decision_id`. Acceptance does not change the deck by itself: any deck
change happens only through a new deck version plus a decision log entry.

## Candidate summaries

### cand-007 - Any-color land mana-base adjustment

- **Type:** `mana_base_adjustment_candidate` | **Status:** `proposed`
- **Incoming:** City of Brass, Mana Confluence | **Outgoing:** none yet
- **Purpose:** increase any-color land sources for a five-color identity
  whose colored-cast consistency the baseline flags as unverified
  (pressure point 2, open question 2).
- **Evidence basis:** baseline pressure point and open question; stored
  candidate facts for both lands (each taps for any color at a damage/life
  cost); project brief mana-fixing goal.
- **Expected impact:** raises colored_source/fixing_land counts (21 and 12
  of 34 lands today) and access to the 23 colored nonland cards.
- **Tradeoffs:** incremental damage/life cost per activation; a fixed
  100-card deck means a later swap target is unavoidable if advanced; the
  deck's colorless production needs weigh against cutting colorless lands.
- **Risks:** life loss accumulates; the underlying failure mode is untested
  until the open-question simulation runs.
- **Why not actionable:** proposed only — requires testing, a Product Owner
  decision, a decision log entry, and a new deck version before any change.

- **Risks:** depends on 0-1 mana value artifact density holding; adds a
  colorless source before fixing is reviewed.
- **Why not actionable:** proposed only — requires testing, a Product Owner
  decision, a decision log entry, and a new deck version before any change.

### cand-009 - Krark-Clan Ironworks explosiveness engine

- **Type:** `add_candidate` | **Status:** `proposed`
- **Incoming:** Krark-Clan Ironworks | **Outgoing:** none yet
- **Purpose:** artifact-sacrifice burst mana; the deck currently has zero
  `ritual_or_burst_mana` role coverage, and the baseline questions whether
  the sole tagged combo payoff (Aetherflux Reservoir) has support.
- **Evidence basis:** baseline pressure point 0 and open questions 0-1;
  stored candidate facts (sacrifice an artifact: add two colorless mana);
  functional-roles data showing the empty burst-mana role.
- **Expected impact:** first ritual_or_burst_mana coverage, a repeatable
  sacrifice outlet, and mana_engine density feeding payoffs.
- **Tradeoffs:** consumes the deck's own artifacts, competing with
  artifact-count payoffs and Myr board presence; four mana on a 3.0 curve.
- **Risks:** **can pull the deck toward a dedicated combo-engine identity —
  an identity decision reserved to the Product Owner — and may reduce
  casual readability**; its strongest lines depend on combo mapping that
  has not been done, so testing is required.
- **Why not actionable:** proposed only — requires testing, a Product Owner
  decision, a decision log entry, and a new deck version before any change.

## Why outgoing slots are deliberately empty

A fixed 100-card deck means any accepted addition eventually displaces a
card. This set does not nominate cut targets: choosing what leaves the deck
is a Product Owner judgment that should follow, not precede, review of what
comes in. If a candidate advances, a follow-up review names the outgoing
card as part of the decision, and only the resulting decision log entry and
new deck version change the deck.

## Boundary note

This candidate set creates proposed candidates only. It does not approve,
reject, or implement any candidate. It does not edit the deck, create a new
version, write a decision, alter card facts or candidate card facts, alter
Knowledge data, change baseline analysis, or modify rec-001.
No deck change is authorized.
