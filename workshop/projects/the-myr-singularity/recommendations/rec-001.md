# Recommendation Candidate Schema — rec-001

## What this file is

This document (with its machine-readable companion `rec-001.json`) defines
the **schema** for future recommendation candidates for The Myr Singularity.
It is not a recommendation. It contains zero candidates, names no cards,
and authorizes no deck change of any kind.

The Workshop's product rule is: Knowledge first. Reasoning second.
Recommendation third. Card Facts, Functional Roles, and the Baseline
Structural Analysis v1.0 now exist; this task defines the contract that any
future recommendation must satisfy before it can even be proposed.

## Why traceability is required

A recommendation candidate is only admissible if it can cite its lineage:

- **Baseline analysis** — every candidate must reference the `analysis_id`
  that motivated it, plus the specific pressure points and/or open questions
  it responds to (referenced by index, e.g.
  `baseline_v1.0:pressure_point:0`).
- **Project goals** — every candidate must serve at least one stated goal
  from the project brief (explosiveness, consistency, mana fixing,
  resilience, Myr identity preservation). Candidates serving no stated goal
  are out of scope by definition.
- **Card facts** — any card a candidate touches is referenced by Scryfall ID
  from `workshop/card-data/cards.json`, never by unverified names.
- **Functional roles** — structural claims must point at role and category
  IDs from `workshop/knowledge/role_taxonomy.json`, so that expected impact
  is measurable against the same distributions the baseline reports.

This keeps recommendations auditable: a reviewer can follow every candidate
back to evidence, and a validator can mechanically reject candidates with
broken references.

## Candidate lifecycle

```
proposed → under_review → accepted | rejected | deferred | needs_testing
                ↑                                    │
                └──────────── (with new evidence) ───┘
```

- New candidates start as `proposed`.
- `under_review` candidates resolve to `accepted`, `rejected`, `deferred`,
  or `needs_testing`.
- `needs_testing` and `deferred` candidates return to `under_review` —
  `needs_testing` only with new playtest or simulation evidence attached.
- `accepted` and `rejected` are terminal and require a recorded
  `user_decision` and `decision_id` (a decision log entry in `decisions/`).
- **Acceptance does not change the deck.** Any deck change happens only
  through a new deck version file plus a decision log entry. Existing
  version files are never edited.

## Candidate fields

Each future candidate record has six field groups (full contract with
types, required flags, and allowed values is in `rec-001.json`):

- **Identity** — `candidate_id` (e.g. `cand-001`), `candidate_type`
  (`add_candidate`, `cut_candidate`, `swap_candidate`, `package_candidate`,
  `mana_base_adjustment_candidate`, `role_coverage_candidate`,
  `knowledge_review_candidate`), and lifecycle `status`.
- **Traceability** — `source_analysis_id`, `source_pressure_point_ids`,
  `related_open_question_ids`, `related_project_goals`, `related_roles`,
  `related_categories`.
- **Evidence** — `evidence_summary`, `evidence_items` (each typed as
  `baseline_analysis`, `card_facts`, `functional_roles`, `project_brief`,
  `playtest_note`, `simulation_result`, or `user_preference`), and
  `confidence` (`low`/`medium`/`high` — curator confidence, not a card
  quality score).
- **Content** — `proposed_change`, `affected_cards`, `expected_impact`,
  `tradeoffs`, `risks`, `constraints_checked`.
- **Boundary** — `is_actionable` (always false until a user decision),
  `requires_user_decision`, `requires_testing`, `creates_new_deck_version`,
  `decision_log_required`.
- **Review** — `reviewer_notes`, `user_decision`, `decision_id`,
  `follow_up_tasks`.

Two candidate types deliberately do not propose deck changes:
`role_coverage_candidate` describes a role-level gap without naming cards,
and `knowledge_review_candidate` proposes reviewing Knowledge coverage
(for example, the baseline's observation that no card carries a
win-condition role may be a tagging gap rather than a deck gap).

## What a future Task 16 may do

A later task may populate `candidates` with actual candidate records that
conform to this schema — for example, candidates responding to the
baseline's pressure points. Doing so requires: set `status` beyond
`schema_only`, keep every candidate `proposed` and non-actionable, satisfy
every rule in `validation_expectations`, and leave all decisions to the
Product Owner. Extending the validator to enforce this schema mechanically
is also natural follow-up work.

## Boundary note

No deck changes are authorized by this file. It contains no card cuts, no
card adds, no swaps, no named cards, no upgrade/budget/price content, no
popularity claims, and no decisions. The `candidates` array is empty and
must remain empty while `status` is `schema_only`.
