# First Internal Recommendation Candidate Set - rec-001

## What this file is

This document (with its machine-readable companion `rec-001.json`) records the
first internal recommendation candidate set for The Myr Singularity. The
candidates are limited to cards already present in the current Card Facts layer
and are inputs for Product Owner review only.

Every candidate is `proposed`, non-actionable, and undecided. No deck change is
authorized by this file. Any future deck change requires an explicit Product
Owner decision, a decision log entry, and a new deck version file.

## Evidence boundary

The candidates trace to stored Sprint 1 artifacts:

- **Baseline analysis** - pressure points and open questions from
  `analysis/baseline_v1.0.json`.
- **Project goals** - goals from the project brief and project metadata.
- **Card facts** - affected cards are referenced in JSON by Scryfall ID from
  `workshop/card-data/cards.json`.
- **Functional roles** - role and category IDs come from
  `workshop/knowledge/role_taxonomy.json`.

Human-readable summaries may name cards. Machine-readable `affected_cards`
entries use `scryfall:<id>` references.

## External candidate limitation

This PR intentionally does not propose external add or swap cards. The current
validator requires every `affected_cards` reference to resolve against
`workshop/card-data/cards.json`, and the current Card Facts layer contains only
the deck and sideboard card facts.

External card additions require a future Candidate Card Facts Intake task.
Outside candidate cards must be imported into Card Facts before they can be
referenced in `affected_cards`.

## Candidate lifecycle

```text
proposed -> under_review -> accepted | rejected | deferred | needs_testing
                ^                                    |
                +------------ with new evidence -----+
```

- New candidates start as `proposed`.
- `accepted` and `rejected` are terminal and require `user_decision` and
  `decision_id`.
- Acceptance does not change the deck by itself.
- Any deck change happens only through a new deck version plus a decision log.
